# LangGraph Agent Dashboard

A production-style LangGraph workflow for a support-ticket agent with state management, conditional routing, retry loops, human-in-the-loop approval, persistence, and metrics. Ships with a React + Vite frontend dashboard and a FastAPI backend.

---

## Prerequisites

- **Python** 3.10+
- **Node.js** 18+ (for the frontend)
- **pip** or **conda**

---

## Quick Setup

### 1. Clone & install Python dependencies

```bash
# Option A: conda
conda create -n langgraph-agent python=3.11
conda activate langgraph-agent
pip install -e '.[dev]'

# Option B: venv
python -m venv .venv
# Windows: .venv\Scripts\activate
# macOS/Linux: source .venv/bin/activate
pip install -e '.[dev]'

# With SQLite persistence support (recommended)
pip install -e '.[dev,sqlite]'
```

### 2. Install frontend dependencies

```bash
cd frontend
npm install
```

### 3. Start the backend API

```bash
make run-api
# → Backend running at http://localhost:8000
```

### 4. Start the frontend dev server

```bash
cd frontend
npm run dev
# → Frontend running at http://localhost:5173
```

The frontend proxies `/api/*` requests to the backend on port 8000. Start the backend first, then the frontend.

## Running the Scenarios

```bash
# Run all scenarios and generate metrics
make run-scenarios

# Validate the output
make grade-local
```

This generates `outputs/metrics.json` with per-scenario results.

## How Routing Works

The `classify_node` assigns every query to one of five routes using **keyword priority** (highest first):

| Priority | Route | Keywords (examples) |
|---|---|---|
| 1 | `risky` | refund, delete, cancel, terminate, send, revoke, suspend, ban |
| 2 | `tool` | status, order, lookup, find, check, track, retrieve |
| 3 | `missing_info` | Vague / short queries with placeholders ("it", "this", "that") |
| 4 | `error` | timeout, fail, error, crash, unavailable, exception |
| 5 | `simple` | Everything else (how-to, FAQ, informational) |

**Note**: Risky keywords are checked first. A query like "Refund and track my order" routes to `risky` (highest priority), not `tool`.

## State Schema

Append-only audit fields (use `Annotated[list, add]`):

| Field | Reducer | Purpose |
|---|---|---|
| `messages` | append | Conversation audit log |
| `tool_results` | append | Tool call record |
| `errors` | append | Error log |
| `events` | append | Per-node LabEvent log |

Overwrite fields: `route`, `risk_level`, `attempt`, `final_answer`, `proposed_action`, `approval`, `evaluation_result`

| Category | Points | What we look for |
|---|---|---:|
| Architecture & state schema | 20 | Typed state with correct reducers, lean serializable fields |
| Graph behavior | 25 | All routes correct, bounded retry loop, HITL path, all terminate |
| Persistence & recovery | 15 | Checkpointer wired, thread_id per run, crash-resume evidence |
| Metrics & tests | 20 | `metrics.json` valid, scenario coverage, tests pass |
| Report & demo | 15 | Architecture explanation, metrics table, failure analysis |
| Production hygiene | 5 | Config, env handling, lint/type discipline |

**Grade bands**: 90+ (production + extension), 75–89 (core works), 60–74 (incomplete), < 60 (broken).

## Common Pitfalls

1. **Keyword conflicts**: "Check order status" has both "check" and "order" (both tool keywords). Risky keywords always win over tool keywords.

2. **Unbounded retry**: Always check `attempt < max_attempts`. Without this bound, error scenarios loop forever.

3. **SqliteSaver API**: Use `SqliteSaver(conn=sqlite3.connect(...))` — not `from_conn_string()` which returns a context manager.

4. **Forgetting finalize**: Every route must reach `finalize → END`. Missing this means the graph hangs.

5. **Word boundary matching**: "Can you fix it?" — match "it" as a whole word, not as a substring of "item" or "iteration". Strip punctuation before checking.
